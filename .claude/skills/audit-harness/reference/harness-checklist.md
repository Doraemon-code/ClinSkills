# harness-checklist.md

`audit-harness` 的评判标准。11 个维度分两组：**跨文件维度**（需纵览全局，主 skill 完成）
与**单文件维度**（逐 artifact 判断，可按需 spawn）。附文件体量软上限与严重度定义。

> **范围**：只评 **git 追踪的**文件；被 gitignore 的 `CLAUDE.md`/`config.*`/`04 scripts/`
> 与 `~/.claude/` 下的 Memory 不受审（至多读作上下文）。

---

## 一、跨文件维度（主扫全局）

### 1. 一致性/漂移
文档与实际是否脱节。

- CLAUDE.md / SKILL.md / rule 里描述的**路径、文件名、函数名、字段、sheet 名**是否仍存在。
- Memory 或注释里点名的 file/function/flag 是否还在（recall 内容反映的是写入当时的状态）。
- 多个 skill/rule 之间的**约定是否冲突**（同一件事两处说法不一）。
- CLAUDE.md 里的表格/清单（如超窗分类、Agent 清单）是否与 `.claude/` 实际内容同步。

### 2. 冗余/重复
同一信息在多处维护 = 漂移隐患。

- 同一规则/路径/阈值是否在 SKILL.md、constraints.md、CLAUDE.md 重复硬编码。
- 是否存在「本应单一来源、却复制多份」的内容——建议留一处权威、其余引用。
- 判据：重复本身不必然是问题，但**会各自独立漂移**的重复必须消除。

### 3. 覆盖缺口
该有而没有的。

- 高频/高风险操作是否缺护栏（hook / permission deny / rule）。
- 是否缺应有的 skill 或 agent（反复手工做的事没有自动化）。
- 现有 skill 的流程是否有未覆盖的分支（如某 EDC 类型、某异常输入无处理路径）。

## 二、单文件维度（逐 artifact，可 spawn）

### 4. 提示词质量
skill/agent 的可触发性与可执行性。

- **description**：触发词是否精准、覆盖用户可能的自然语言说法；是否 ≤ 500 字符
  （过长反而降低触发判定精度）；是否清楚说明「何时触发」。
- **步骤**：是否清晰、无歧义、可被 LLM 直接执行；有无「靠猜」的空档。
- agent system prompt：职责是否单一、返回格式是否明确。

### 5. 护栏健壮性
拦截是否真的拦得住。

- hook / permission deny 规则是否**真能**拦下目标操作（如 raw 数据读取、危险命令），
  有无绕过路径（别名、等价命令、大小写、相对路径）。
- 约束是否同时有「行为指引 + 机制拦截」双保险（命令行不支持 hooks 时的降级）。
- 失败模式是否安全（拦截失败时是 fail-closed 还是 fail-open）。

### 6. 最佳实践符合度
对照 Claude Code skill/agent 编写惯例。

- 一个 skill 单一职责，触发边界清晰、不与其他 skill 抢触发。
- 命名、目录结构（reference/library/scripts 分层）符合既有约定。
- frontmatter 完整、格式正确。
- 无 `~/.claude/CLAUDE.md` 里禁止的反模式（过度设计、投机抽象、无关重构）。

### 7. 稳定性/体量
文件过长会稀释指令、挤占任务上下文、降低执行稳定性。**软上限**（提示拆分，非硬报错）：

| 文件类型 | 软上限 | 说明 |
|---|---|---|
| **SKILL.md** | **≤ 200 行** | 触发即整体入上下文，最该精瘦；超出下沉到 reference |
| description（frontmatter） | ≤ 500 字符 | 触发判定用 |
| reference/*.md（单文件） | ≤ 300 行 | 按需加载，但须单一职责，超出即应拆 |
| library / scripts *.py | ≤ 350 行 | 超出考虑拆模块 |
| agent *.md | ≤ 150 行 | system prompt 过长稀释指令 |
| hook *.py | ≤ 150 行 | 单一拦截职责 |

达软上限 **80%** 即标为「关注」，超上限标为「问题」。判断时结合职责：
单一职责但偏长 < 多职责但未超线。

### 8. 引用合理性
文件间指路是否可靠。

- SKILL.md 引用的 reference/library 路径是否存在、拼写正确。
- INDEX.md、CLAUDE.md 的 Agent 清单等索引是否与实际文件对应。
- 引用是否有效降低主文件体量（而非只是把内容搬走却仍在 SKILL.md 复述一遍）。
- 循环引用 / 悬空引用（指向已删除文件）。

### 9. 渐进式披露（Progressive Disclosure）
按需加载而非一次性全给。

- SKILL.md 只留「触发即需要」的核心流程；细则、清单、长模板下沉到 reference，用到才读。
- 大段代码模板、编码规范、校验清单不应内联在 SKILL.md。
- 分层是否合理：SKILL.md（流程）→ reference（标准/知识）→ library/scripts（可执行件）。
- 反模式：SKILL.md 里塞满只在特定分支才用到的详细内容。

### 10. 泛化性 / 可复用性
harness 能否复用到同类新项目，而非绑定当前项目的专属假设。**本维度为审计核心之一。**

- 逻辑体内是否**硬编码项目专属字面量**：具体列名 / sheet 名 / 量表名 / 访视名、某语言的
  取值（如中文 `"是"` / `"其他"`）、只对某一 EDC 或某一场景成立的分支。
- 尤其警惕**散落在 `@config` / 参数块之外**的硬编码——复用者不会注意到需要改，且失败方式
  常是**静默漏检 / 错误结果**而非报错，对泛化伤害最大（如配置暴露了灵活性、解析器却写死类型）。
- 修法：提为 config 常量 / 函数参数（**保留当前值作默认以向后兼容**），或下沉到调用方。
- 模板 / 骨架（skeleton、library 模板）是否残留具体项目内容；示例是否明确标「示例」，
  而非把专属工具叙述成通用能力。
- 判据：通用工具替调用方决定业务文案 / 单位 / 语言 = 泛化性缺陷。

### 11. 简洁性 / 死代码
最小实现、无冗余。对齐 `~/.claude/CLAUDE.md` 的 Simplicity First。**审计应直接建议删除。**

- **死代码**：未使用的 import / 变量 / 函数 / 分支；计算后从不被读取的中间量；no-op 调用
  （如列名已就位仍恒等 rename）。删掉后行为不变的代码一律删。
- **过度设计**：投机式可配置性、单处调用的包装函数、为不可能场景写的错误处理、冗余的双 gate。
- **注释 / docstring 漂移**：描述与实际实现不符（写的是未实现的意图）——订正或删除。
- 判据：能 50 行别写 200 行；每一处保留都应有当下用途。

## 严重度定义

| 严重度 | 含义 |
|--------|------|
| 严重 | 护栏失效导致约束可被绕过、引用悬空致 skill 直接跑不通、约定冲突致行为不可预测 |
| 高 | 文档与实际严重漂移、触发词失准致 skill 触发不了、明显违反最佳实践 |
| 中 | 冗余漂移风险、体量超软上限、渐进式披露不佳、覆盖缺口 |
| 低 | 命名/风格/可读性小瑕疵、体量逼近但未超软上限 |

## 输出与执行约定

- 无问题的维度**不在表格占行**；全维度无问题时明确写「未发现明显问题」并逐维标「通过」。
- 位置写精确锚点：行号（`L120-179`）、frontmatter 字段名、或表格名。
- 问题描述写「当前状态」，改法写在建议列。
- 报告阶段只读不改；修改在 SKILL.md 第 5 步交互式逐条进行。
