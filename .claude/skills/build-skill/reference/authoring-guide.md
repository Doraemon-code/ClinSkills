# skill 编写判据（authoring-guide）

编写 Claude 能可靠发现并使用的 skill 的完整标准。SKILL.md 是概览与流程，本文件是可按需展开的判据。

## 目录
- 核心原则：简洁、自由度、多模型
- description 写法
- 命名约定
- 渐进式披露
- 工作流与反馈循环
- 内容指南
- 常见模式（模板 / 示例 / 条件分支）
- 含代码的 skill
- 反模式

## 核心原则

**简洁至上。** 上下文窗口是公共资源，SKILL.md 一旦触发即整体入上下文，每 token 都与对话历史竞争。默认假设 **Claude 已很聪明**，只补它不知道的：对每条信息问「Claude 真需要这解释吗？能否假设它已知？值这些 token 吗？」——别解释 PDF 是什么、库怎么用这类常识。

**自由度匹配任务脆弱性**（把 Claude 想成探路机器人）：
- **开阔地**（多路径都成功）→ 高自由度：文本指令给大方向，信任 Claude 找最优。例：代码审查。
- **有首选模式** → 中自由度：伪代码或带参脚本，允许变化。例：报表生成。
- **两侧悬崖的窄桥**（唯一安全路径、须一致、须定序）→ 低自由度：精确脚本 +「勿改此命令」。例：数据库迁移。

**多模型测试**：skill 是模型的补充，用你将用的所有模型测（Haiku 需更多细节，Opus 忌过度解释）。

## description 写法

`description` 决定发现——Claude 靠它从可能上百个 skill 里选中正确的那个。
- **第三人称**（会注入系统提示，人称不一致伤发现）：✓ "Processes Excel files…" ✗ "I can…" / "You can…"。
- 同时写**做什么 + 何时用**，含具体触发关键词 / 场景。
- ≤1024 字符（硬上限），仍力求精简。
- 反例：`Helps with documents` / `Processes data`（太泛、选不中）。
- 例：`Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDFs, forms, or document extraction.`

## 命名约定

- **动名词优先**：`processing-pdfs`、`analyzing-spreadsheets`（清楚描述能力）。
- 可接受：名词短语 `pdf-processing`、动作式 `process-pdfs`。
- 规则：≤64 字符，仅小写字母 / 数字 / 连字符；忌保留词 `claude`/`anthropic`。
- 避免：`helper`/`utils`/`tools`（太泛）、集合内命名风格不一致。

## 渐进式披露

三级加载：级别 1 元数据（always）→ 级别 2 SKILL.md（触发时）→ 级别 3 reference/scripts（引用时；脚本执行不入上下文）。三种内容类型各展所长：**指令**（md，灵活指导）、**代码**（script，可靠确定）、**资源**（md/数据，事实查询）。

- SKILL.md 只留「触发即需要」的核心流程；细则、长清单、大模板下沉 reference。
- **引用只一层深**：所有 reference 从 SKILL.md 直接链接。嵌套（A→B→C）会被 `head -100` 部分预览、信息不全。
- reference `*.md` **>100 行加目录（TOC）**，部分读取时也能看清全貌。
- 组织模式：① 高层指南 + 引用；② 按领域分文件（只载相关领域）；③ 条件详情（基本内容内联、高级链接）。

## 工作流与反馈循环

- **复杂任务给清单**：让 Claude 复制到回复逐项勾选，防跳步。
- **反馈循环**（验证器 → 修错 → 重复）显著提质。验证器可是脚本，也可是参考文档（读它自查）。
- **计划-验证-执行**：高风险 / 批量操作先产结构化计划文件 → 脚本验证 → 再执行；错误早发现、计划可回滚、调试清晰。

## 内容指南

- **无时效信息**：会过时的（"2025-08 前用旧 API"）放「旧模式」折叠区（`<details>`），不混入主流程。
- **术语一致**：一个概念一个词（别混 endpoint/URL/path、field/box/element、extract/pull/get）。

## 常见模式

- **模板模式**：给输出格式模板。严格要求用「ALWAYS 用此结构」；灵活时给「合理默认 + 按需调整」。
- **示例模式**：输出质量依赖看示例时，给 input/output 对，比纯描述更能传达风格与详略。
- **条件工作流**：决策点分支（"新建？→ 创建流程；编辑？→ 编辑流程"）；分支庞大就拆到单独文件按需读。

## 含代码的 skill

- **解决问题，不推卸**：脚本显式处理错误（文件不存在→建默认；无权限→给替代），别 `open(path).read()` 直接崩、让 Claude 收拾。
- **无魔法常量**（Ousterhout 定律）：每个配置值加注释说明理由——你都不知正确值，Claude 更不知。
- **提供实用脚本**：预制脚本比现场生成更可靠、省 token、更一致。明确是**执行**（"运行 analyze.py 提字段"）还是**作参考读**（"参见 analyze.py 的算法"）——多数用执行。
- **显式依赖**：SKILL.md 列所需包，别假设已装（`pip install pypdf` 后再用）；Claude API 运行时无网络、装不了包。
- **MCP 工具用全限定名**：`ServerName:tool_name`，否则多服务器下可能「找不到工具」。

## 反模式

- Windows 反斜杠路径 → 一律正斜杠（`scripts/x.py`），跨平台才不报错。
- 堆砌多个等价方案 → 给一个默认 + 逃生舱（"扫描件需 OCR 时改用 …"）。
- 假设包已装 → 显式列依赖。
- SKILL.md 塞满只在特定分支才用的细则 → 下沉 reference。
- 深层嵌套引用 → 保持一层深。
